### US-005: Harden security and operational limits for web terminal control
Status: in_progress
Depends on: US-003

Description:
As a platform owner, I want secure and bounded terminal control so console access cannot execute unauthorized or abusive command traffic.

Acceptance Criteria:
- [ ] All attach/command/detach handlers enforce existing authorization and reject unauthenticated calls before runtime resolution.
- [ ] Add per-session command rate limiting and max command length validation with explicit API error codes.
- [ ] Audit records include whether actions were accepted or rejected and include rejection reason when blocked.
- [ ] Example: authorized operator issues valid command within limits and receives accepted execution path.
- [ ] Negative case: unauthorized request (missing/invalid bearer token) receives HTTP 401 and produces no command execution in runtime pod.
- [ ] Negative case: rapid burst exceeding rate policy returns throttling response and records throttle reason in audit metadata.
