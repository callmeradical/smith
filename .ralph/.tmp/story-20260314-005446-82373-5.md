### US-005: Block non-ready PRDs during ingress and execution handoff
Status: in_progress
Depends on: US-001, US-002, US-004

Description:
As a Smith operator, I want invalid or non-ready PRDs blocked before ingress or loop execution so Ralph loops do not start from malformed requirements.

Acceptance Criteria:
- [ ] Update PRD ingress and any PRD-to-loop handoff path to call the shared validation core before creating loop drafts or writing workspace PRD artifacts.
- [ ] Return structured diagnostics to callers when the PRD fails readiness validation, including suggested fixes when available.
- [ ] Example: a valid canonical PRD posted through `POST /v1/ingress/prd` is accepted and creates loop drafts with the expected metadata.
- [ ] Negative case: a PRD with oversized stories, missing quality gates, or malformed dependencies is rejected before loop creation and no runtime job is scheduled from that PRD.
- [ ] Integration tests cover successful ingress, validation rejection, and parity between CLI validation results and API rejection reasons.
