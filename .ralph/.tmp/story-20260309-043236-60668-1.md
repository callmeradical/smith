### US-001: Resolve runtime pod/container for a loop
Status: in_progress
Depends on: None

Description:
As an operator, I want the API to resolve the current runtime pod/container for a loop so the console can determine whether terminal attach is possible.

Acceptance Criteria:
- [ ] Add runtime resolution logic that maps loop ID to namespace, pod name, container name, and attachability status.
- [ ] Expose `GET /v1/loops/{id}/runtime` with explicit fields for `attachable` and human-readable `reason` when false.
- [ ] Example: for an active loop with Running replica pod, endpoint returns `attachable=true` with concrete pod and container names.
- [ ] Negative case: for a terminal loop or missing pod, endpoint returns `attachable=false` with reason `loop not active` or `runtime pod not found`.
- [ ] Unit tests cover running, pending, terminal, and missing-runtime scenarios.
