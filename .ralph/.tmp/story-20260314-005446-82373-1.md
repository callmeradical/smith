### US-001: Define canonical PRD validation and diagnostic contracts
Status: in_progress
Depends on: None

Description:
As a Smith maintainer, I want a shared canonical PRD validation contract so CLI and API workflows can evaluate the same document structure and produce the same diagnostics.

Acceptance Criteria:
- [ ] Add or extend a shared PRD validation package that validates required top-level fields, story IDs, dependency references, quality gates, and canonical status expectations.
- [ ] Add a machine-readable validation report structure with stable diagnostic codes, field or story references, and optional suggested fixes.
- [ ] Example: a canonical JSON PRD with sequential `US-001`, `US-002` stories and valid dependencies returns `valid=true` with zero blocking errors.
- [ ] Negative case: a PRD with a missing project name, duplicate story IDs, or dependency on unknown `US-999` returns blocking diagnostics tied to the failing fields or stories.
- [ ] Unit tests cover valid documents, malformed documents, duplicate IDs, bad dependencies, and missing quality gates.
